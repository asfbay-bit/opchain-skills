## Workers Deployment

### Project Structure (Hono + D1)

```
my-app/
├── src/
│   ├── index.ts          # Hono app entry, lifespan-like setup
│   ├── routes/           # Route modules
│   ├── db/
│   │   ├── schema.ts     # Drizzle schema (source of truth)
│   │   └── migrations/   # Generated by drizzle-kit
│   ├── middleware/        # Auth, CORS, error handling, logging
│   ├── lib/              # Shared utilities
│   └── types.ts          # Env bindings type
├── frontend/             # React app (Vite, deployed to Pages)
├── wrangler.toml
├── drizzle.config.ts
└── vitest.config.ts
```

### Env Bindings Type

```typescript
// src/types.ts — typed bindings for Workers
export interface Env {
  DB: D1Database;
  CACHE: KVNamespace;
  AUTH_SECRET: string;     // wrangler secret
  ENVIRONMENT: string;     // "production" | "staging" | "development"
}
```

### No Cold Starts, No Connection Pooling

Unlike Cloud Run, Workers have:
- **Zero cold starts** — V8 isolates start in <5ms
- **No connection pool math** — D1 bindings handle connections automatically
- **No lifespan management** — no startup/shutdown hooks needed
- **No instance scaling config** — Workers scale automatically per-request

This eliminates Sections 4 and 5 of the enterprise guide entirely.

## D1 Patterns

### Drizzle ORM Setup

```typescript
// src/db/index.ts
import { drizzle } from "drizzle-orm/d1";
import * as schema from "./schema";

export function getDb(env: Env) {
  return drizzle(env.DB, { schema });
}
```

### Query Patterns

```typescript
// Explicit eager loading — no lazy loading footgun with D1
const usersWithOrders = await db
  .select()
  .from(users)
  .leftJoin(orders, eq(orders.userId, users.id))
  .where(eq(users.id, userId));

// Batch queries for related data (D1 equivalent of selectinload)
const userList = await db.select().from(users);
const userIds = userList.map(u => u.id);
const orderList = await db
  .select()
  .from(orders)
  .where(inArray(orders.userId, userIds));
```

### Migrations

```typescript
// drizzle.config.ts
import type { Config } from "drizzle-kit";

export default {
  schema: "./src/db/schema.ts",
  out: "./src/db/migrations",
  driver: "d1-http",
  dialect: "sqlite",
  dbCredentials: {
    accountId: process.env.CF_ACCOUNT_ID!,
    databaseId: process.env.D1_DATABASE_ID!,
    token: process.env.CF_API_TOKEN!,
  },
} satisfies Config;
```

```bash
# Generate migration
npx drizzle-kit generate

# Apply locally
npx wrangler d1 migrations apply MY_DB --local

# Apply to production
npx wrangler d1 migrations apply MY_DB --remote
```

**Always review generated migrations.** Drizzle Kit (like Alembic) detects renames as
drop + add. Manual edit to `ALTER TABLE ... RENAME COLUMN` when appropriate.

### D1 Limitations to Know

- Max 10GB per database (more than enough for aidops-scale)
- No full-text search (use KV or an external search service)
- No `LISTEN/NOTIFY` (use Queues or polling for real-time)
- SQLite semantics (type affinity, not strict typing — Drizzle handles this)
- Write throughput: ~1K writes/sec sustained (batch inserts help)

## KV Patterns

Use KV for: caching, session storage, feature flags, config, static page content.
Do NOT use KV for: primary data storage, frequently updated data, transactional operations.

```typescript
// Cache pattern with TTL
async function getCached<T>(kv: KVNamespace, key: string, ttl: number, fetcher: () => Promise<T>): Promise<T> {
  const cached = await kv.get(key, "json");
  if (cached) return cached as T;

  const fresh = await fetcher();
  await kv.put(key, JSON.stringify(fresh), { expirationTtl: ttl });
  return fresh;
}

// Usage
const user = await getCached(env.CACHE, `user:${id}`, 300, () => getDb(env).select().from(users).where(eq(users.id, id)));
```

### KV for Page Registry (aidops pattern)

```typescript
// Registry: KV key = route path, value = page config
await env.PAGES_REGISTRY.put("cycle-tracker", JSON.stringify({
  title: "CycleTracker",
  workerUrl: "https://cycle-tracker.aidops.dev",
  status: "active",
}));
```

## Pages for Frontend

### Deploy React SPA to Pages

```bash
# Build frontend
cd frontend && npm run build

# Deploy to Pages
npx wrangler pages deploy frontend/dist --project-name=my-app
```

### Custom Domain Routing

```
aidops.dev           → Pages (landing/homepage)
app.aidops.dev       → Pages (React SPA)
api.aidops.dev       → Worker (API)
```

Or single domain with path-based routing via a Worker:
```
aidops.dev/app/*     → Pages
aidops.dev/api/*     → Worker
```

### SPA Fallback

Pages handles SPA routing automatically — all paths serve `index.html` when no static
file matches. No server-side configuration needed.

## Hybrid Architecture

When you need Python backend (FastAPI) but want CF for frontend and edge:

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  CF Pages    │────▶│  CF Worker   │────▶│  FastAPI      │
│  (React SPA) │     │  (API proxy) │     │  (Cloud Run / │
│              │     │  + KV cache  │     │   Fly.io)     │
└──────────────┘     └──────────────┘     └──────────────┘
                            │
                     ┌──────────────┐
                     │  D1 / KV     │
                     │  (edge data) │
                     └──────────────┘
```

The Worker acts as a BFF proxy: handles auth, caching, rate limiting, and CORS at the
edge, forwarding to FastAPI for business logic. This gives you Python's ecosystem with
CF's edge performance.

## Wrangler Configuration

### Minimal wrangler.toml

```toml
name = "my-app-api"
main = "src/index.ts"
compatibility_date = "2025-05-01"

[[d1_databases]]
binding = "DB"
database_name = "my-app-db"
database_id = "your-database-id"

[[kv_namespaces]]
binding = "CACHE"
id = "your-kv-id"

[vars]
ENVIRONMENT = "production"
```

### Environments (staging vs production)

```toml
# Production (default)
[vars]
ENVIRONMENT = "production"

# Staging override
[env.staging]
name = "my-app-api-staging"
[env.staging.vars]
ENVIRONMENT = "staging"

[[env.staging.d1_databases]]
binding = "DB"
database_name = "my-app-db-staging"
database_id = "staging-database-id"
```

```bash
# Deploy to staging
npx wrangler deploy --env staging

# Deploy to production
npx wrangler deploy
```

### Secrets

```bash
# Set a secret
npx wrangler secret put AUTH_SECRET
# Prompts for value, never stored in wrangler.toml

# List secrets
npx wrangler secret list
```

## CI/CD Pipeline

### GitHub Actions: Full Deploy Pipeline

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  CLOUDFLARE_API_TOKEN: ${{ secrets.CF_API_TOKEN }}
  CLOUDFLARE_ACCOUNT_ID: ${{ secrets.CF_ACCOUNT_ID }}

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20" }
      - run: npm ci
      - run: npx vitest run
      - run: npx tsc --noEmit

  deploy-api:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20" }
      - run: npm ci
      # Run migrations BEFORE deploying new code
      - run: npx wrangler d1 migrations apply MY_DB --remote
      - run: npx wrangler deploy

  deploy-frontend:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20" }
      - run: cd frontend && npm ci && npm run build
      - run: npx wrangler pages deploy frontend/dist --project-name=my-app

  deploy-staging:
    needs: test
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: "20" }
      - run: npm ci
      - run: npx wrangler d1 migrations apply MY_DB_STAGING --remote
      - run: npx wrangler deploy --env staging
```

### Deploy Order

1. Run migrations (schema changes before code that depends on them)
2. Deploy API Worker
3. Deploy frontend Pages

This matches the enterprise guide's "deploy migrations before code" principle.

## Cost Model

### Cloudflare Free Tier (covers most aidops-scale apps)

| Service | Free Allowance | Paid Rate |
|---|---|---|
| Workers | 100K requests/day | $0.30/million requests |
| D1 | 5M rows read, 100K writes/day | $0.001/million rows read |
| KV | 100K reads, 1K writes/day | $0.50/million reads |
| Pages | Unlimited static hosting | — |
| R2 | 10GB storage, 10M reads/month | $0.015/GB/month |

**Realistic cost for a 2-user app like CycleTracker:** $0/month on free tier.
**Realistic cost for a 100-user internal tool:** $0-5/month.
**Breakpoint where costs matter:** ~1M+ requests/day, ~50M+ D1 reads/day.

Compare to Cloud Run: minimum ~$3.24/month for a single always-on instance,
plus Cloud SQL (~$7+/month for the smallest instance). CF is dramatically cheaper
at small scale.
